# WhoWare 3D Pivot — Implementation Plan

After the initial 2D panorama release, the next step is to make the daily
memories explorable as actual 3D scenes. The player walks through a
memory, finds clues on real 3D props, and the AI-generated scene brief
drives prop placement and lighting.

The plan is split into phases. Each phase ships a usable, end-to-end
build — there are no half-step "scaffolding-only" deliveries.

## Principles

These guide every change in this plan:

- **ENHANCEMENT FIRST** — extend existing components (PanoramaScene,
  SceneCanvas) over creating new ones
- **CONSOLIDATION** — delete dead code rather than deprecate it
- **PREVENT BLOAT** — adopt the threejs-game-skills pack as a
  reference, not as a copy-paste. Don't add infrastructure we
  can't exercise today
- **DRY** — the AI scene brief is the single source of truth for
  what a scene *is*; both 2D and 3D renderers consume it
- **CLEAN** — 3D is a self-contained module under
  `components/who-ware/scene-3d/` with explicit dependencies
- **MODULAR** — skybox, props, lighting, controls are independently
  testable
- **PERFORMANT** — adaptive: 3D is opt-in per device, falls through
  to 2D seamlessly
- **ORGANIZED** — new domain folder mirrors the existing
  `who-ware/` structure

## What's shipped (Phases 0-2)

The current state of the 3D pivot, in chronological order:

- **Phase 0 — Seam.** Three.js added as a dependency
  (`apps/default/package.json`); `scene-quality.ts` does capability
  detection; `MemoryScene` is the orchestrator that picks the
  renderer; `SceneCanvas` is a placeholder. No behaviour change at
  the default-mode level.

- **Phase 1 — Skybox.** The existing AI panorama image is mapped
  onto the inside of a sphere; the player can drag to look around.
  Clue hotspots from the existing `x/y` screen positions project
  onto the sphere's inner surface via
  `hotspotWorldPosition()` (covered by unit tests). 5 skybox math
  tests passing.

- **Phase 2 — Props + lighting.** Scene schema extended with
  `SceneProp[]` (closed vocabulary of 51 kinds) and `SceneLighting`.
  `prop-registry.ts` is the single source of truth on the server.
  `prop-shapes.ts` builds procedural primitive compositions for
  every kind. `lighting-rig.ts` builds the three-point lighting rig.
  `SceneCanvas` now composes skybox + lighting + props; click
  detection raycasts the scene and routes to the bound prop.

### Phase 2.1 — Adaptive 2D/3D + resilience (shipped after the plan was written)

- **Adaptive loader.** `scene-quality.ts` reads `localStorage` for a
  user override and the GPU renderer string for low-power patterns
  (`SwiftShader`, `llvmpipe`). Falls through to the 2D
  `PanoramaScene` silently — no new infra.
- **Lazy 3D bundle.** `MemoryScene` lazy-imports `SceneCanvas` via
  `Suspense`, so 2D-mode users never download Three.js.
- **Crash containment.** `components/shared/error-boundary.tsx`
  wraps the 3D scene + leaderboard + on-chain overlay. A
  render-phase crash shows a "Reload scene" button instead of
  white-screening the entire game (and the player's streak).
- **Timeout hygiene.** The transition-layer `setTimeout` chain
  stores IDs in a ref and clears them on unmount — no more
  state-update-on-unmounted-component warnings during fast scene
  cycling.

### Phase 3 — AI-driven prop generation (1-2 weeks)

The procedural primitives in `prop-shapes.ts` are deliberately crude —
a desk is a box on four legs, a globe is a sphere on a cylinder. They
serve as the always-works fallback. Phase 3 replaces hero props with
real GLB meshes generated by Tripo.

- **Tripo API integration** for `hero` props (a flag in the registry).
  Triggered at curation time, cached in Convex storage, served
  alongside the panorama image.
- **Asset budget** — GLBs ≤ 200kb each, textures ≤ 1MB. Hero props
  limited to one per scene to keep download cost low.
- **Graceful fallback** — if Tripo fails, the scene renders the
  procedural primitive. Pipeline never blocks on hero-prop
  generation.
- **Per-prop env var** `TRIPO_API_KEY`; the registry entry declares
  whether a kind is "hero" (Tripo-eligible) or "primitive-only"
  (procedural forever, e.g. floor / wall).

### Phase 4 — Movement & interaction (1 week)

Phase 1+2 gives the player look-around + click. Phase 4 gives them
feet.

- **First-person controls** — drag to look, WASD / on-screen
  joystick to walk. Pitch-clamp from `look-controls.ts` carries over.
- **Simple collision** — bounding-box check against prop positions,
  no physics engine. WhoWare doesn't need rigid bodies.
- **Inspection hover** — pointing at a clickable prop (within
  ~3 units) shows a soft outline + label. The label is the
  prop's `clueLabel` (if any) so the player knows what they're
  about to find.
- **Native support** — add `expo-three` + `expo-gl` for iOS / Android.
  Web-only stays the default for the first release.

### Phase 5 — QA + adaptive performance (1 week)

- **Capability matrix** in `scene-quality.ts`:
  - WebGL2 + dpr ≤ 2 → three-d, full LOD
  - WebGL2 + dpr > 2 → three-d, internal LOD
  - WebGL1 or no WebGL → panorama
- **Performance budget** — 60fps on 2020 MacBook Pro, 30fps on
  iPhone 12. Adaptive LOD drops poly count if `performance.now()`
  deltas exceed budget.
- **Visual regression** — `agent-browser` snapshots of the 3D
  scene at canonical camera angles, diffed against committed
  baselines. Existing tooling; no new infra.
- **Telemetry** — track 3D-mode usage, fallback rate, FPS samples
  (privacy-preserving, no PII).

### Phase 6 — Polish & rollout (3-4 days)

- **Settings toggle** — let users pick 2D / 3D / auto per device.
- **Onboarding video** — 15-second clip showing the 3D experience.
- **Archive paywall unchanged** — payment still gates closed
  episodes in both modes. Same 402 flow, same `PAYWALL_TREASURY_ADDRESS`.
- **Reduced-motion fallback** — if the user has prefers-reduced-motion
  on, default to 2D mode.
- **Accessibility** — keyboard nav for movement, captions for the
  ambient text, audio description of the scene as a stretch.

## What's explicitly NOT in scope

- **No new AI provider dependencies beyond Tripo** — keep Venice +
  Replicate. Tripo behind a feature flag.
- **No physics engine** — no Rapier, no cannon-es. We don't need
  rigid bodies. Per `threejs-gameplay-systems` skill guidance,
  those are for "serious" games; WhoWare's "movement + click" model
  is simpler.
- **No real-time multiplayer** — single-player daily ritual.
- **No mobile-first 3D push** — web-only at first; mobile stays
  2D until Phase 5 proves out.
- **No NPC characters** — the historical figure is never shown,
  per the AI scene brief's "no portraits" rule.

## What we're taking from the threejs-game-skills pack

The pack at `github.com/majidmanzarpour/threejs-game-skills` is a
useful reference for patterns. We adopt:

| Skill | What we use |
|---|---|
| `threejs-gameplay-systems` | Architecture boundaries (`core`/`game`/`entities`/`systems`/`assets`/`ui`/`tests`) → our `scene-3d/` folder structure |
| `threejs-aaa-graphics-builder` | Lighting presets and post-processing pass list → `LightingRig.tsx` |
| `threejs-3d-generator` | Tripo API integration pattern → Phase 3 hero props |
| `threejs-image-generator` | Asset pipeline concept (cache generated assets, fall back to procedural) → Phase 3 GLB caching |
| `threejs-debug-profiler` | FPS sample collection, adaptive LOD → Phase 5 `quality.ts` |
| `threejs-qa-release` | QA gate list (build, browser, screenshots, canvas pixels, mobile viewport) → new `scripts/qa-scene-3d.ts` |

We **don't** take:

- The Vite + TypeScript scaffold (WhoWare is on Expo + Metro; two
  toolchains would create a maintenance burden).
- The full file layout — we mirror our existing conventions, not
  the pack's.
- Rapier / cannon-es physics — see "NOT in scope" above.

## Open decisions for future

1. **First-device default** — web-only at first, or attempt mobile
   at Phase 4? (Recommended: web-only until Phase 5 proves out.)
2. **Hero prop source** — Tripo or procedural primitives only?
   (Recommended: procedural first, Tripo behind a feature flag for
   the most photogenic scenes.)
3. **Public asset hosting** — committed GLBs in `assets/` or
   Convex storage generated at curation time? (Recommended:
   committed for hand-authored hero props; Convex storage for
   AI-generated.)
4. **Bundle budget for first paint** — 5MB cap on the 3D path OK,
   or stricter?

## File map (where the 3D code lives)

```
apps/default/components/who-ware/
├── panorama-scene.tsx                # existing 2D renderer (fallback)
├── memory-scene.tsx                  # orchestrator: picks 3D vs 2D
└── scene-3d/
    ├── SceneCanvas.tsx               # <canvas> + WebGLRenderer + RAF loop
    ├── skybox.ts                     # equirectangular sphere + texture loader
    ├── look-controls.ts              # pointer-driven yaw/pitch
    ├── prop-shapes.ts                # procedural primitive compositions
    ├── lighting-rig.ts               # three-point ambient + key + fill
    └── (Phase 3+) prop-assets.ts    # GLB loading + caching

apps/default/lib/
└── scene-quality.ts                  # capability detection (WebGL2, dpr, override)

packages/backend/convex/
├── schema.ts                         # scene + props + lighting
├── catalog.ts                        # AI scene brief prompt + saveSceneBriefs
├── props.ts                          # prop vocabulary (single source of truth)
├── props.test.ts                     # registry tests
└── _generated/api.d.ts               # Convex-generated bindings
```
